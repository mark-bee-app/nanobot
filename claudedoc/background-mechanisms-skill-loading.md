# 后台机制中的 Skill 加载策略

在 nanobot 的架构设计中，**Heartbeat（心跳机制）**、**Cron（定时任务机制）** 以及 **Dream（记忆整合机制）** 都扮演着重要的异步 / 后台角色。但因为它们的目标和上下文深度不同，加载 `skill` 的策略也存在显著差异。

以下是它们在 Skill 加载上的具体表现及异同对比：

## 1. 各个机制是如何加载 Skill 的？

### **Heartbeat（心跳机制）**
- **Phase 1（唤醒决策阶段）**：**完全不加载 Skill**。仅提供一段极简的 System Prompt 以及 `HEARTBEAT.md` 文件的内容，通过虚拟的工具回调让大模型判断有没有任务要做。
- **Phase 2（任务执行阶段）**：如果判定需要执行，会通过底层的 `AgentLoop.process_direct` 走完整的一轮 Agent 对话。此时加载 Skill 的方式与处理用户消息**完全一致**：
  - 调用 `ContextBuilder` 和 `SkillsLoader`。
  - 对于 Frontmatter 中被标记为 `always=true` 的技能（且满足运行环境要求），会将它的**完整内容**内联到 Prompt 的 `# Active Skills` 块中。
  - 对于其他的技能，只会生成一个名为“技能摘要”的列表（包含：名称、描述、文件路径），大模型想要使用时必须通过 `read_file` 工具读取具体内容。

### **Cron（定时任务机制）**
- 定时任务触发（如用户的日程提醒、周期性脚本等）时，其内部实际上调用了 `AgentLoop.process_direct`。
- 因此，常规的定时任务**在 Skill 的加载逻辑上与 Heartbeat Phase 2 完全相同**：会走完整的 `ContextBuilder` 构建流程，全文注入 `always=true` 的技能，对其他技能则仅提供清单和路径。

### **Dream（记忆整合机制）**
- **Phase 1（历史梳理阶段）**：为了精简 Token 并专注于记忆整理，**不加载任何 Skill 信息**。
- **Phase 2（文件编辑落盘阶段）**：Dream 需要把总结好的知识写入文件（有时需要写入新的 Skill 文件）。这个阶段跳过了 `ContextBuilder`，直接使用 `AgentRunner`，加载 Skill 的方式极为克制：
  - 调用专用的 `_list_existing_skills()` 函数扫描 `skills/` 和系统内置技能目录。
  - 它通过非常简单的正则去匹配每个 `SKILL.md` 里的 `description` 字段。
  - 最后它仅仅会拼接出一个如 `[技能名] — [技能描述]` 的**纯文本小列表**，放在 Prompt 的 `## Existing Skills` 标题下。
  - **不会直接展开（内联）任何技能的具体内容**。哪怕该技能标记了 `always=true`，它也不会被自动放入上下文中。如果模型需要看现有技能的模板，必须使用挂载的文件读取工具（`ReadFileTool`）自行去读。

---

## 2. 有什么是相同的？

1. **统一的技能数据源**：无论是哪种机制，读取到的技能来源都是完全一样的，即工作区目录下的 `skills/` 目录以及系统的自带目录 `BUILTIN_SKILLS_DIR`。
2. **都给模型提供了“能力大纲（上帝视角）”**：这三种机制在执行“真正的工作”时（Heartbeat 和 Cron 的日常工作，Dream 阶段 2 的文件写入），都会在提示词里告诉大模型当前系统里存在哪些技能以及它们的作用。这样可以防止大模型无中生有，或者反复“发明”系统内已经存在的功能。

---

## 3. 有什么是不同的？

1. **加载深度的不同（最核心的差异）**
   - **Heartbeat & Cron** 在干活时更看重**开箱即用**。它们遵循完整的上下文逻辑，会自动解包并全文载入高频使用的技能（`always=true`），大模型可以直接根据 Prompt 中的规则执行复杂任务。
   - **Dream** 在干活时为了防止 Prompt 被业务技能撑爆干扰到长记忆和代码写入的注意力，采用了**只读目录**模式。它提供技能列表的唯一目的是作为“知识去重”的参考，防止它总结记忆时创建了一个系统里早就有的技能。
2. **挂载工具的不同**
   - **Heartbeat & Cron**：会挂载整个 Agent 的全部可用工具（如 `exec` / `search` / `web_fetch` 等等），大模型拿到技能就能立刻联动各类工具干活。
   - **Dream**：在阶段 2 只会为 AgentRunner 挂载最基础的三个纯文本 IO 工具：`ReadFileTool`，`WriteFileTool`，`EditFileTool`。哪怕技能详情里要求大模型执行 Shell 脚本，此时也是执行不了的，这也是为什么它根本不需要去加载那些“业务技能”完整规则的原因。
