# Nanobot 记忆系统核心架构逻辑

Nanobot 的记忆系统核心逻辑集中在 `nanobot/agent/memory.py` 中。它被设计为三个核心层次，分别处理不同生命周期的记忆数据：

## 1. `MemoryStore` —— 纯文件 I/O 存储层
这是最底层的数据持久化结构，负责所有记忆文件的读写操作。

*   **职责**：
    *   维护工作区（workspace）下的文件系统，主要是 `memory/` 目录。
    *   管理长期的系统事实记忆（`MEMORY.md`）。
    *   管理角色设定（`SOUL.md`）和用户偏好（`USER.md`）。
    *   追加并读取对话历史记录（`history.jsonl`），并维护一个递增的游标（`.cursor` 和 `.dream_cursor`）。
*   **关键机制**：
    *   **追加历史 (`append_history`)**：在每次对话后，它会清理大模型的“思考过程”（`strip_think`），然后将内容安全地追加到 JSONL 文件中，并赋予一个全局递增的 `cursor`（游标）。
    *   **兼容性**：包含一次性的迁移逻辑，负责将老版本的 `HISTORY.md` 迁移为结构化的 `history.jsonl`。

## 2. `Consolidator` —— 轻量级上下文压缩器（短期记忆整理）
这是在主 Agent Loop（对话循环）中同步执行的模块，用于保证大模型的上下文不会爆炸。

*   **职责**：
    *   监控当前会话的 Token 数量（`maybe_consolidate_by_tokens`）。
    *   当历史消息过多，快要触及模型的上下文窗口上限（`context_window_tokens`）时，触发归档。
*   **工作流程**：
    1.  选取一个安全的边界（不能切断一个完整的用户-助手对话回合）。
    2.  调用大模型，将这段多轮的历史对话总结成一段摘要（Summary）。
    3.  将摘要通过 `MemoryStore` 写入历史文件。
    4.  从内存中丢弃这些原始消息，将下一次提问时的上下文替换为这段摘要。
    5.  如果 LLM 总结失败，系统有降级机制（`raw_archive`），会将原始文本粗暴截断存档，以确保程序继续运行。

## 3. `Dream` —— 重量级的异步记忆处理器（长期记忆沉淀）
这个模块设计用来在后台（通常是 cron 定时任务）执行，它像人类“做梦”一样，将短期的零散对话提炼成长期记忆。

*   **职责**：
    *   分析 `history.jsonl` 中尚未处理的新对话记录。
    *   提取出有价值的新事实、用户偏好或系统状态，并更新到 `MEMORY.md`、`SOUL.md` 和 `USER.md` 中。
*   **两阶段工作流 (Two-phase)**：
    *   **Phase 1（分析）**：读取历史记录和当前的记忆文件状态，调用 LLM 产生一份“更新建议分析”。这里有一个巧妙的设计：它会使用 Git Blame 标注 `MEMORY.md` 中每一行的“年龄（Age）”（例如 `← 30d` 表示 30 天未改动），帮助 LLM 识别哪些记忆已经过时需要清理。
    *   **Phase 2（执行）**：启动一个带有专门工具（`read_file`, `edit_file`）的 `AgentRunner` 子代理。将第一阶段的分析结果交给子代理，让它自主去精确编辑 `MEMORY.md` 等文件。
*   **最终落地**：如果编辑成功，它会将游标（`dream_cursor`）向前推进，并通过 `GitStore` 自动 commit 这一批次的记忆变更。

---

## 总结：它们是如何协同工作的？

1.  **聊天时**：用户与 Nanobot 交流，产生的新消息不断被 `MemoryStore` 追加到 `history.jsonl` 中。
2.  **聊得太多时**：`Consolidator` 会在后台计算 Token。一旦超标，它会将早期的几轮对话总结成一段摘要，释放 Token 空间，保证对话流畅。
3.  **休息时（Cron 任务）**：`Dream` 模块在后台启动。它沿着 `dream_cursor` 找到所有新的聊天记录，利用 LLM 进行反思，将其中有价值的经验、用户偏好抽取出来，通过工具自动修改 `MEMORY.md` 或 `USER.md`。修改完成后，自动 git commit。下次聊天时，这些提炼出的长期记忆就会被注入到 System Prompt 中。
