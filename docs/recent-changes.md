# 近期变更

本文档记录那些尚未完全反映到稳定版用户指南、或值得单独留档的重要设计与行为变更。

## 2026-06-18 — Dream 不再编辑 `SOUL.md`

**提交：** `8e3b647d` — *dream机制不去修改soul*

### 背景

`SOUL.md` 最初被视为 Dream 可以读取并编辑的长期记忆文件之一。Dream 是记忆系统的第二阶段整合器，负责把 `memory/history.jsonl` 蒸馏成持久知识。

### 决策

Dream 现在**有意忽略** `SOUL.md`，只读取和更新：

- `USER.md` — 用户身份、偏好与稳定的用户事实
- `memory/MEMORY.md` — 项目上下文、事实与持久知识

`SOUL.md`（机器人格、语气与沟通风格）保留给用户或代理显式编写，不再由 Dream 自动维护。

### 改动内容

- 从 Dream 第一阶段提示词上下文中移除 `SOUL.md`。
- 从 Dream 第二阶段可编辑文件列表中移除 `SOUL.md`。
- 在 `nanobot/agent/memory.py` 中删除 `_SOUL_FILE_MAX_CHARS` 截断常量。
- 更新 `nanobot/templates/agent/dream_phase1.md` 和 `dream_phase2.md`，仅提及 `USER.md` 和 `MEMORY.md`。
- 更新 `tests/agent/test_dream.py` 中的断言，使其反映只有 `USER.md` 被视为永久文件且不带年龄标记。

### 原因

- 人格应该是克制的、由人主导的，而不是从会话摘要中自动涌现出来的。
- 让 `SOUL.md` 保持用户可编辑，能让机器人语气更可预测、更可审计。
- 防止 Dream 在长期运行中悄然改变机器人的沟通风格。

### 影响

- `/dream` 仍会按计划和手动触发运行。
- `/dream-log` 和 `/dream-restore` 仍可用于查看和恢复记忆变更。
- 如果之前依赖 Dream 自动维护 `SOUL.md`，现在需要显式编辑该文件。

## 2026-06-13 — `/git` 指令简化为单一同步工作流

**提交：** `2340e6a5` — *更新 Git 命令处理器以支持工作区路径参数*  
**提交：** `3f84ffe0` — *简化内置命令中的 Git 操作功能*

### 背景

`/git` 内置命令原先暴露多个子命令：`status`、`diff`、`log`、`add`、`commit`、`pull`、`push`、`sync`。这虽然提供了细粒度控制，但增加了复杂度，实际聊天中也很少用到除“全部同步”之外的流程。

### 决策

将多子命令设计替换为一个高层次的单一同步工作流。`/git` 现在只做一件事：暂存所有变更、提交、拉取、再推送。

### 改动内容

- 保留：
  - `/git [message]` — 执行同步工作流
  - `/git status` — 显示工作区状态
  - `/git log [n]` — 显示最近 n 条提交（默认 10，最大 50）
  - `/git help` — 显示帮助文本
- 移除的子命令：
  - `diff`
  - `add`
  - `commit`
  - `pull`
  - `push`
  - `sync`
- 在 `nanobot/command/builtin.py` 中新增 `_git_sync_workflow`，执行顺序为：
  1. 检查工作区状态；
  2. 暂存并提交所有变更（使用用户提供的提交信息，默认为 `sync: update from nanobot`）；
  3. 从远端拉取；
  4. 检测拉取后是否产生合并冲突；
  5. 若无冲突则推送到远端。
- 新增冲突检测：如果 `git pull` 引入冲突，则跳过 push 步骤，并提示用户先解决冲突。
- 更新帮助文本，描述新的同步工作流。
- 清理未使用的导入并简化异常处理。

### 原因

- 最常见的使用场景是“全部同步”，其他子命令在聊天中很少单独使用。
- 单一工作流减少了命令表面，避免用户陷入中间状态。
- 内置冲突检测能让用户得到清晰的下一步提示。

### 影响

- 之前使用 `/git diff`、`/git add` 等命令的脚本或用户，需要改用原生 `git` 命令或新的 `/git [message]` 工作流。
- 未提供提交信息时，默认使用 `sync: update from nanobot`。
- 该工作流针对配置的工作区运行，并会暂存**所有**变更，而不是某个子集。
